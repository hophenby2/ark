## 1. 核对方式

可以直接编辑本文的“用户结论”和“证据”行，也可以按下面的格式回复：

```text
Q01=A，主基线使用 Android a14b4a；Windows 作为兼容基线。
Q03=部分确认：gold 先领，附件见 /path/to/capture。
Q05=B，在拿到 Boss 样本前维持零矩阵收益。
```

状态统一使用：

- `待核对`：尚未收到结论。
- `已确认`：有同版本表、协议样本或可复核资料。
- `被否定`：当前猜测与证据冲突。
- `保守方案`：暂时无证据，明确采用不发奖、不推进或返回不支持的安全行为。

每条结论最好同时记录：确认日期、适用客户端/Data 版本、证据文件或链接。真实协议样本需要去除账号、token、设备 ID 等敏感字段。

## 2. 已确定口径

以下内容已有明确决定，不需要再次核对：

- 运行时不访问网络；外部资料只能生成带来源和版本的离线规则。
- SQLite Repository 是当前局、随机种子和 revision 的唯一真源；非法动作必须事务回滚。
- 未知 theme、zone、node type 不得 clamp，也不得回退到普通/紧急收益。
- 普通/紧急基础收益使用 [rlv2_logic.py](../server/rlv2_logic.py) 的 60 格固定矩阵；`rogue_4/5` 仅显式允许 `zone 7 -> 第六档`。
- 地图硬验收是每个受支持 `theme x zone` 组合 2500 个固定种子；10000 个仅作发布前非阻塞分布审计。
- 无证据的概率、奖励池和乘区必须标记为近似或保持关闭，不能伪装成原版精确值。

- 当前 `roguelike_topic_table.json` SHA-256 为 `643df7574c8955c827bec2645ed09c06df44bc6654a85ead96002a8298b91bb6`。

疑问：后续抓包和 fixture 应以 Android、Windows，还是两者同时为基线？两份资源是否有影响 RLV2 的热更差异？

- **A（推荐）**：每份 fixture 使用 `clientVersion + dataVersion + topicTableHash + platform` 作为版本键，指定一个主基线，另一个只做兼容回归。
- **B**：暂时只记录 `2.7.51 / 26-07-10`，忽略平台和完整 Data 版本。成本低，但可能把不同热更样本混在一起。

### Q03：当前战斗奖励协议假设是否正确？

已知事实：

- `RoguelikeStageEarn` 元数据只有 `exp/populationMax/squadCapacity/hp/shield/maxHpUp`，没有 gold。
- 当前实现让 EXP 在 `battleFinish` 自动入账和升级，gold 作为独立 `RoguelikeReward` 领取。
- 当前响应 reward `index` 为字符串、`done` 为布尔值；选择请求要求整数 `index + sub`。
- 当前全职业券仍是占位奖励，基础 gold 未领取时禁止 `finishBattleReward`。

疑问：gold 是否必须手动领取、奖励顺序和 index 是否正确、未领 gold 是否允许结束、`exDrop/exDropSrc` 如何出现、全职业券是否必掉？

候选方案：

- **A（推荐）**：抓普通与紧急战斗各一条完整的 `battleFinish -> chooseBattleReward -> finishBattleReward`，对响应做 exact fixture。
- **B**：证据前维持当前兼容外形，但只标记为协议假设；所有外形字段集中在适配层，便于后续替换。

需要核对：普通/紧急响应、逐次领取请求、每步 current 变化，以及跳过券/跳过 gold 的客户端行为。

用户结论：`待填写`

证据：`待填写`

疑问：每个主题、楼层、stage/variant 的 EXP、gold、券和藏品是什么；减半作用于哪些项目，如何舍入？

需要核对：每主题至少一个 Boss 的节点对象、stage ID、战斗结算、奖励领取，以及有/无减半条件的对照。

### Q09：事件楼层资料是否可以作为版本化离线事实？

已知事实：PRTS 页面可映射 RO1/2/4/5 的事件楼层；RO3 有重名和 month 变体，需要消歧；当前运行时会从主题全部 enter 场景随机选取。

- **A（推荐）**：生成含 `source/revision/theme/floors/modes` 的离线映射，启动时强校验场景引用；未映射条目进入 quarantine 且默认不可出现。
- **B（不建议）**：运行时按名称/前缀猜测，或未匹配时回退全池。该方案容易提前出现结局事件。

需要核对：是否接受对应 PRTS revision、RO3 重名映射表，以及 quarantine 中歧义项的人工结论。

用户结论：`待填写`

需要核对：优先事件清单、进入条件、choice 前后状态、一次性/跨层行为，以及无证据事件的期望 UI 行为。

## 8. 状态、安全与部署问题

### Q13：状态转移矩阵以抓包为准还是先按现有流程收紧？

已知事实：多个 handler 分散处理 `pending`，有队列和栈混用；缺少集中式状态不变量。

疑问：真实客户端允许哪些 pending 嵌套、跳过、重试和乱序；不同非法动作使用什么错误码？

候选方案：

- **A（推荐）**：先按已经测试成功的主流程建立保守 command 矩阵，未知顺序返回 409/422 且状态不变，再用完整会话放宽。
- **B**：等五主题完整会话齐备后再重构。风险是现有宽松 handler 继续积累状态污染。

需要核对：战斗奖励插入招募、跳过奖励、事件战斗、失败重试等动作序列和错误响应。

用户结论：`待填写`

证据：`待填写`

### Q14：客户端是否提供可用的幂等键？

已知事实：revision/CAS 防止陈旧覆盖，但不能让重复请求返回第一次结果。

疑问：客户端请求是否包含稳定的 requestId、seq 或 header；断网重试时是否复用？

候选方案：

- **A（推荐）**：使用客户端稳定键，记录 `(run_id, request_id, command, result_revision/result)`。
- **B**：若没有原生键，使用 `(uid, run_id, input_revision, route, canonical body hash)` 的短窗口指纹；必须定义碰撞和过期行为。
